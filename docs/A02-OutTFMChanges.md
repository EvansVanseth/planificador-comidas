# Captura de inconsistenticas/errores
Como inicio de la arquitectura y cuando se planteó el DDD. Se estimó oportuno capturar los errores o estados inválidos lanzando excepciones (throw) que las capas superiores o elementos superiores (entities de VO, o aggregates de entities) capturan de los inferiores al realizar acciones. Pero se ha visto que este patrón crea una obligatoriedad de duplicación de comprobaciones en la capa más externa, la UI o el servicio BackEnd o lo que se tenga implementado de la capa de aplicación hacia arriba. Se requiere muchas veces la intervención o testeo directo por parte de esta capa de elementos que sería mas correcto que lo hiciese la capa de aplicación. Por ese motivo uno de los cambios a plantear es la modificación del patrón de captura de errores. En lugar de lanzar excepciones lo que haremos es devolver siempre un objeto, que indica el exito o fallo de la operación y que adjunta los datos correctos en el caso del éxito y una lista de errores en el caso de fallo. Lo vemos más simplio y la UI tiene más control sobre lo que mostrar al usuario cuando se produce un error, aunque siempre podrá mostrarse directamente el mensage que contesta la operación de igual forma que ahora se muestra el mensage del error.
