# Especificación de Requisitos de Software (SRS) - Planificador de Comidas Semanal

Este documento formaliza los requisitos del sistema derivados del Documento de Requisitos de Producto (PRD) original (`First approach.md`). Define de manera inequívoca qué debe hacer el sistema (Requisitos Funcionales) y bajo qué restricciones de calidad debe operar (Requisitos No Funcionales).

---

## 1. Requisitos Funcionales (RF)

Los requisitos funcionales definen los servicios, acciones y comportamientos que el sistema debe ejecutar obligatoriamente.

### Módulo 1: Gestión de Recetas (Recipes Management)
*   **RF-01.1: Registro de Recetas Únicas:** El sistema debe permitir la creación, lectura, actualización y eliminación (CRUD) de recetas. El nombre de la receta actuará como identificador único para cada usuario.
*   **RF-01.2: Parámetros Base de Receta:** Cada receta debe almacenar de forma obligatoria:
    *   Nombre (único por usuario).
    *   Cantidad base de comensales (número entero positivo).
    *   Tiempo aproximado de elaboración (en minutos).
*   **RF-01.3: Gestión de Ingredientes:** Cada receta debe contener un listado de ingredientes. El sistema debe sugerir autocompletado en base a ingredientes preexistentes en la base de datos para evitar duplicidades de nomenclatura.
*   **RF-01.4: Campo Instrucciones de Elaboración:** Cada receta debe admitir un campo de texto enriquecido u opcional con la descripción paso a paso de la elaboración de la receta (con carácter puramente informativo por el momento).
*   **RF-01.5: Clasificación mediante Metadatos (Etiquetas):** Las recetas deben ser clasificadas mediante cuatro dimensiones taxonómicas de etiquetas:
    *   **Momento del día (Meal Time):** Desayuno, Comida, Cena (Selección obligatoria de al menos uno; admite selección múltiple).
    *   **Formato de Servicio (Format):** Caliente, Frío (Selección obligatoria de al menos uno; admite selección múltiple).
    *   **Tipo de plato (Dish Type):** Carne, Pescado, Legumbres, Pasta, Ensalada, Arroz, Dulce (Selección obligatoria de al menos uno; admite selección múltiple).
    *   **Estilos de vida/Dieta (Dietary Labels):** Bajo en calorías, Vegetariano, Vegano, Alérgenos (especificando tipo), Personalizadas (libre creación). (Opcional, admite selección múltiple).
*   **RF-01.6: Catálogo Maestro de Ingredientes:** El usuario debe poder visualizar el listado completo de sus ingredientes maestros registrados y poder editarlos o eliminarlos de forma independiente.

### Módulo 2: Planificador de Menús (Planner Engine)
*   **RF-02.1: Inicialización de Planificación:** El sistema debe permitir crear una planificación definiendo:
    *   Nombre descriptivo (opcional, útil para identificar plantillas o períodos especiales).
    *   Fecha de inicio (obligatoriamente debe ser un lunes del calendario, u opcional si actúa como plantilla).
    *   Cantidad de semanas a planificar (número entero positivo).
*   **RF-02.2: Estructura Diaria de Planificación:** El sistema generará una matriz de días basada en las semanas seleccionadas. Cada día debe permitir configurar de forma independiente para cada servicio (Desayuno, Comida, Cena):
    *   Número de comensales específicos (si se establece en 0, el servicio queda excluido de la planificación).
    *   Listado de etiquetas de exclusión (etiquetas que el motor no debe incluir).
    *   Listado de etiquetas de preferencia (etiquetas que el motor priorizará incluir).
*   **RF-02.3: Edición Manual y Sobrescritura:** El sistema debe permitir al usuario asignar recetas a servicios y días concretos de forma manual antes de ejecutar la planificación automática, respetando esta selección manual durante el proceso de auto-planificación.
*   **RF-02.4: Edición en Lote (Bulk Editing):** El sistema debe proporcionar herramientas de selección múltiple de días (incluyendo un selector rápido para la semana completa) para aplicar configuraciones de comensales, exclusiones y preferencias simultáneamente.
*   **RF-02.5: Duplicación de Planificaciones:** El usuario debe poder duplicar o copiar cualquier planificación existente para crear una nueva (sirviendo como plantilla), clonando el conjunto completo de días relativos con sus asignaciones, exclusiones y preferencias, pero permitiendo definir o posponer la nueva fecha de inicio o cambiar su nombre descriptivo.
*   **RF-02.6: Filtros e Histórico de Planificaciones:** El usuario debe poder acceder a un listado histórico de sus planificaciones y filtrarlas o buscarlas por nombre, fecha de inicio, número de semanas o estado (borrador/activas).

### Módulo 3: Motor de Planificación Automática (Auto-Scheduler Engine)
*   **RF-03.1: Resolución de Restricciones del Servicio:** El motor debe planificar únicamente los servicios del día cuyo número de comensales sea superior a 0.
*   **RF-03.2: Algoritmo de No Repetición:** El motor de planificación debe evitar la repetición de una misma receta y, en la medida de lo posible, de recetas con el mismo "Tipo de plato" dentro del rango temporal planificado.
*   **RF-03.3: Balance de Formato (Frío / Caliente):** El sistema debe proveer un selector de tendencia de formato representado en un rango porcentual del 0% (exclusivamente frío) al 100% (exclusivamente caliente). El motor debe distribuir las recetas calientes y frías de forma intercalada a lo largo del periodo, evitando agrupaciones consecutivas de un mismo formato.

### Módulo 4: Gestión de Despensa e Ingredientes Derivados
*   **RF-04.1: Cálculo Dinámico de Ingredientes:** El sistema debe calcular y consolidar de forma automática la lista total de ingredientes requeridos a partir de la planificación activa.
*   **RF-04.2: Agrupación y Ordenación:** La lista resultante de ingredientes debe presentarse agrupada por ingrediente y ordenada alfabéticamente.
*   **RF-04.3: Estimación de Volumen de Compra:** Para cada ingrediente se debe calcular y mostrar el número total de comensales/comidas acumuladas que lo requieren, ofreciendo una métrica indirecta de volumen.
*   **RF-04.4: Control de Stock en Despensa (Pantry Control):** El usuario debe poder indicar la cantidad de un ingrediente específico que ya posee en su despensa, o marcar el ingrediente por completo con la opción de "Disponer de todo" (tengo de todo).

### Módulo 5: Generador de Lista de la Compra (Shopping List Generator)
*   **RF-05.1: Deducción de Lista de la Compra:** El sistema debe generar la lista de la compra de forma automática restando de la lista de ingredientes necesarios aquellos marcados como disponibles en la despensa (pantry stock).
*   **RF-05.2: Trazabilidad de Compra (Checkout Checklist):** Cada ítem en la lista de la compra debe contar con un control de estado interactivo ("comprado" / "pendiente") para permitir el seguimiento físico de la compra.

### Módulo 6: Seguridad, Autenticación y Multitenencia (Security & Multitenancy)
*   **RF-06.1: Autenticación de Usuarios:** El sistema debe requerir autenticación para permitir el acceso a las funciones de administración de recetas, planificación y listas de compra. Se admitirá registro/login mediante correo electrónico y contraseña.
*   **RF-06.2: Aislamiento estricto de Datos (Multitenancy):** Toda la información transaccional y de catálogo (recetas, ingredientes maestros, etiquetas personalizadas, planificaciones, despensa y checklists) debe estar asociada estrictamente al identificador de usuario (`user_id`). El sistema debe denegar cualquier petición de lectura o escritura sobre registros que pertenezcan a otros identificadores de usuario.

## 2. Requisitos No Funcionales (RNF)

Los requisitos no funcionales definen los atributos de calidad, restricciones técnicas y estándares de experiencia que el sistema debe cumplir.

*   **RNF-01: Usabilidad e Interfaz Adaptativa Dual (Usability & Responsive Layout):** La interfaz de usuario debe ser adaptativa y optimizarse para dos contextos de uso diferenciados:
    *   **Mobile-First (Alta Densidad y Táctil):** Para movilidad en supermercados (lista de la compra) y cocina, simplificando clics, maximizando la legibilidad en pantallas verticales táctiles y garantizando áreas de interacción holgadas para un manejo cómodo a una sola mano.
    *   **Desktop-First (Matriz y Grilla Amplia):** Para pantallas de ordenador de escritorio, ofreciendo una vista panorámica (tipo grilla o matriz de planificación semanal) que facilite la configuración de comensales, exclusiones y preferencias con mayor cantidad de datos simultáneos en pantalla.
*   **RNF-02: Eficiencia del Algoritmo (Performance - Scheduler):** La ejecución del motor de planificación automática para un periodo estándar de hasta 4 semanas debe procesarse y resolverse en un tiempo inferior a 3 segundos bajo condiciones normales de red.
*   **RNF-03: Consistencia y Validación de Datos (Data Integrity):** El sistema debe garantizar que las exclusiones parametrizadas por el usuario sean respetadas de manera estricta (hard constraint), invalidando cualquier asignación automática que las viole.
*   **RNF-04: Seguridad, Encriptación y Robustez (Security & OWASP Compliance):**
    *   **Cifrado y Credenciales:** Toda transferencia de datos debe estar cifrada mediante protocolo HTTPS/TLS. Las contraseñas deben estar protegidas bajo algoritmos de hashing unidireccional criptográficamente fuertes (gestionados por Supabase Auth).
    *   **Protección contra Fuerza Bruta:** Se limitará la tasa de peticiones (*Rate Limiting*) en endpoints críticos (login, signup, recuperación/cambio de claves) para mitigar ataques automatizados de adivinación de contraseñas.
    *   **Prevención de XSS:** Todo texto de entrada libre generado por el usuario (nombres de recetas, ingredientes, comentarios) debe ser sanitizado en el backend y escapado de forma segura en el frontend para neutralizar vulnerabilidades de XSS (Cross-Site Scripting).
*   **RNF-05: Accesibilidad Inclusiva Universal (A11Y - WCAG 2.1 AA):** La aplicación web se maquetará de acuerdo con las pautas de accesibilidad para el contenido web WCAG 2.1 nivel AA:
    *   **Tamaño del Objetivo Táctil:** Todo botón e ítem interactivo del checklist de la compra tendrá un tamaño de clic/toque mínimo de **44x44 píxeles** para evitar errores de selección en movilidad.
    *   **Contraste Visual:** Se garantizará una relación de contraste mínima de **4.5:1** en todo el texto del sistema (especialmente en la lista de la compra del supermercado con reflejos de luz).
    *   **Accesibilidad de Lectores de Pantalla:** Se estructurarán las interfaces utilizando semántica HTML correcta, anuncios dinámicos `aria-live="polite"` para cambios de estado interactivos y cumplimiento estricto de las directrices **WAI-ARIA** para comboboxes y selectores.
